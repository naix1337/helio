⚙️ Collector-Logik:
1. Proxmox API fragen → Liste aller LXC + QEMU
2. Mit DB abgleichen: neu? → Agent anlegen + WS broadcast
3. Entfernt? → Agent offline setzen + WS broadcast
4. Status geändert? → Agent-Status updaten
5. Metriken: CPU/Mem von Proxmox API holen → agent_metrics schreiben
6. last_sync updaten
📡 WebSocket Broadcasts:
{ type: 'proxmox_update', connectionId, resources: [...] }
{ type: 'agent_created', agentId, proxmoxRef }
{ type: 'agent_offline', agentId, reason: 'removed_from_proxmox' }

Agent-Erstellungs-Logik

Wie wird aus einem LXC-Container ein Agent?

Agent-ID:     "proxmox-{connection_id}-{type}-{vmid}"
Agent-Name:   Container/VM-Name (z.B. "mailserver")
Tags:         ["proxmox", "lxc", "connection-{name}"]
OS-Info:      Von Proxmox API (os_type, arch, etc.)
Status:       online (running), offline (stopped)
First-Seen:   Zeitpunkt der ersten Erkennung
